### 🧩 풀이 설계 / 막힌 부분

#### 문제 분석

- 시험은 최대 10,000문제, 문제 정답은 5개 중 하나
- O(N) : 완전 탐색, 플로이드, 이중 반복문 불가능
- 문제 간, 수포자 간 상호작용 없음
- 각 문제에 대해 패턴과 정답을 비교하여 점수를 누적

#### 복잡도 계산

```
answers를 한 번 순회
└─ 각 문제마다
   ├─ 1번 수포자 패턴과 비교
   ├─ 2번 수포자 패턴과 비교
   ├─ 3번 수포자 패턴과 비교
```
사람 수 3 * 문제 수 N = O(N)

#### 설계 후보 비교

1. 규칙을 최대한 수식으로 줄이는 방식
2. 패턴을 배열로 두고 하나씩 비교하는 방식

-> 현재 비교할 패턴 3개, 문제 수 최대 10,000이라면, 전체 연산량은 30,000이 됨. 규칙을 줄여도 체감 성능 차이가 없다. 게다가 1번을 제외하고 가독성이 높아지는 규칙도 없음.

> 패턴은 최소 단위 배열로 정의하고, answers를 한 번 순회하며 비교한다.

### 🧠 대략적인 코드 설명
```js
function solution(answers) {
    const scores = [0, 0, 0];
    const p1 = [1, 2, 3, 4, 5];
    const p2 = [2, 1, 2, 3, 2, 4, 2, 5];
    const p3 = [3, 3, 1, 1, 2, 2, 4, 4, 5, 5];
    
    for (let i = 0; i < answers.length; i++){
        if (answers[i] === p1[i % p1.length]) scores[0]++;
        if (answers[i] === p2[i % p2.length]) scores[1]++;
        if (answers[i] === p3[i % p3.length]) scores[2]++;
    }
    
    const max = Math.max(...scores);
    return scores.map((v, i) => v === max ? i + 1 : null)
                 .filter(v => v !== null);
}
```

아래와 같이 일반화도 가능하나, 현재와 같은 구조에서 가독성만 떨어트리고 성능상 이점은 없음

```js
answers.forEach((a, i) => {
    [p1, p2, p3].forEach((p, j) => {
        if (a === p[i % p.length]) scores[j]++;
    });
});
```
