### 🧩 풀이 설계 / 막힌 부분

#### 문제 분석

* 입력 크기: `sizes.length ≤ 10,000`
* 각 원소는 `[w, h]`, 값 범위 `≤ 1,000`
* 즉, O(N²)는 위험: 완전 탐색, 플로이드, 이중 반복문 불가능
* '모든 명함을 수납', '가장 작은 지갑', '회전 시켜 수납' -> 조합 / 순열 / 경우의 수 / 순서 / 경로 / 거리 / 상호작용 (서로 영향)과 관계 없음
* 매 순간 독립적인 선택이 가능함
* 정렬, 단순 반복으로 처리가 가능함

#### 복잡도 계산

* 모든 명함을 한 번씩 순회, 강 명함에서 두 값 비교 + 최대값 갱신
* 시간 복잡도: O(N)
* 공간 복잡도: O(N) 이하

#### 설계 후보 비교

접근 방식 요약                                           | 판단   | 제외/채택 근거                                                                |
| -------------------------------------------------- | ---- | ----------------------------------------------------------------------- |
| 전체 값 중 최댓값을 기준으로, 그 다음 큰 값을 반복 탐색하며 상대값으로 세로 후보 갱신 | ❌ 제외 | 전역 최댓값을 반복적으로 다시 탐색하는 구조로 불필요한 재탐색 발생. 선택이 독립적이지 않고, 최악의 경우 O(N²) 가능    |
| 후보 1에 정렬을 추가하여 큰 값부터 순차 처리                         | ❌ 제외 | 정렬 비용 O(N log N)이 추가되지만, 문제 본질상 정렬이 필요하지 않음. 즉시 확정되는 선택이 없어 효율 개선 효과 없음 |
| 각 명함을 순회하며 (큰 값, 작은 값)으로 정규화 후, 가로/세로 최대값을 각각 갱신   | ✅ 채택 | 각 명함의 선택이 독립적이며, 한 번의 순회로 즉시 결정 가능. 되돌릴 필요 없는 선택이므로 그리디 조건 충족           |
| 명함 간 크기 관계를 비교하며 가로/세로 조건 분기                       | ❌ 제외 | 명함 간 상호 비교를 도입하여 문제를 불필요하게 관계 비교 문제로 변형. 요구사항 대비 과한 설계                  |

### ⭐️ 문제에서 주로 사용한 알고리즘

- **그리디 (Greedy)/ 선형 스캔 (O(N))**
- 이 문제의 유형은 완전 탐색(Brute Force)으로 분류되지만, 실제로는 시간 복잡도에 따라 완전탐색이 불가능하다.
- 사고 방식 측면에서는, 각 명함이 '회전하지 않는 경우', '회전하는 경우' 두 가지 가능성을 가지므로, 모든 회전 가능성을 고려하는 완전 탐색적 사고로 접근하는 것이 맞다.
- 하지만 각 명함의 선택이 서로 영향을 주지 않아, 가능한 경우를 실제로 모두 생성할 필요는 없으며, 각 명함을 독립적으로 즉시 결정할 수 있어 구현은 그리디 + 선형 스캔으로 해결 가능하다.

<br>

### 🧠 대략적인 코드 설명

```js
function solution(sizes) {
    let maxW = 0;
    let maxH = 0;

    for (const i of sizes) {
        const width = Math.max(...i);
        const height = Math.min(...i);

        maxW = Math.max(maxW, width);
        maxH = Math.max(maxH, height);
    }
    return maxW * maxH;
}
```
