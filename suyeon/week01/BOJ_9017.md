### 📖 풀이한 문제
- 백준 9017 - 문제 이름
- 문제 링크: https://www.acmicpc.net/problem/9017
- 난이도: 실버 3

### 🧩 풀이 설계 / 막힌 부분
- 처음 접근 아이디어
팀 순서 저장할 배열에 입력받은거 쭉 넣으면서, 팀에 대한 팀원수도 HashMap에 저장해 - teammatesCount
N만큼 돌면서 teammatesCount가 6인 팀에 대해서만 등수를 매겨. - rank

상위 4명 저장할 HashMap, 5번째 멤버 점수 저장할 HashMap, 현재 팀 별로 몇 명 저장했는지 볼 HashMap 필요함
fourSum, fifth, memberCount
N만큼 돌면서 각 팀 별로 상위 4명 점수 합산하고 (각 팀 별 몇 명 저장했는지 계속 업데이트 해줘야함) 5번째 저장할 때 fifth에 저장

상위4명 기준으로 오름차순할건데, 만약 점수가 같다면 fifth를 기준으로 정렬.
그러니까 정렬 1순위는 상위4명 점수고, 2순위는 5번째 멤버의 점수

정렬된 리스트의 맨 앞 팀이 정답
- 고려한 시간복잡도: O(N)

### ⭐️ 문제에서 주로 사용한 알고리즘
- 구현

### 🧠 대략적인 코드 설명
```java
public class Main {

    public static void main(String[] args) throws Exception {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringBuilder sb = new StringBuilder();

        int T = Integer.parseInt(br.readLine());
        while (T-- > 0) {
            int N = Integer.parseInt(br.readLine());
            String str = br.readLine();
            StringTokenizer st = new StringTokenizer(str);

            int[] teamIds = new int[N];
            HashMap<Integer, Integer> teammatesCount = new HashMap<>();

            for (int i = 0; i < N; i++){
                teamIds[i] = Integer.parseInt(st.nextToken());
                teammatesCount.put(teamIds[i], teammatesCount.getOrDefault(teamIds[i], 0) + 1);
            }

            int rankNum = 1;
            int[] rank = new int[N];
            for (int i = 0; i < N; i++) {
                if (teammatesCount.get(teamIds[i]) == 6){
                    rank[i] = rankNum++;
                }
            }

            HashMap<Integer, Integer> fourSum = new HashMap<>();
            HashMap<Integer, Integer> fifth = new HashMap<>();
            HashMap<Integer, Integer> memberCount = new HashMap<>();

            for (int i = 0; i < N; i++) {
                if (rank[i] == 0){
                    continue;
                }
                int memCount = memberCount.getOrDefault(teamIds[i], 0) + 1;
                if (memCount == 5){
                    fifth.put(teamIds[i], rank[i]);
                } else if (memCount <= 4) {
                    fourSum.put(teamIds[i], fourSum.getOrDefault(teamIds[i], 0) + rank[i]);
                }
                memberCount.put(teamIds[i], memCount);
            }

            ArrayList<Map.Entry<Integer, Integer>> list = new ArrayList<>(fourSum.entrySet());

            list.sort((a,b) -> {
                if (!a.getValue().equals(b.getValue())){
                    return a.getValue().compareTo(b.getValue());
                }

                return fifth.get(a.getKey()).compareTo(fifth.get(b.getKey()));
            });

            sb.append(list.get(0).getKey()).append('\n');

        }
        System.out.println(sb);
    }
}
```

### ⏰ 시간복잡도
`O(N)`
#### 📍 리스트 정렬은 O(MlogM) 아닌가?
>여기서 M은 팀의 갯수이다. M < 200
데이터가 많아져도 정렬해야 할 리스트의 크기는 최대 200을 넘지 않으므로 상수라고 봐도 된다. 따라서 무시 가능.

#### 📍 근데 N도 < 1000 이라는 천장이 있는데 왜 얘는 상수취급을 안하지?
>1️⃣ N은 **입력의 크기**이기 때문이다. 주인공 버프(ㅋㅋ)를 받는다고 생각하면 됨.
>
Big O 표기법에서 변수로 살아남는 기준은 **데이터가 들어오는 통로의 크기**인가 하는 점이다.
- **N**: `readLine()`을 통해 읽어오는 데이터의 개수. 알고리즘의 존재 이유라고 할 수 있음.
- **M (200):** 데이터의 개수가 아니라, 그 데이터가 가질 수 있는 **값의 범위**
>
보통 알고리즘을 분석할 때는 **입력 데이터(N)가 무한히 늘어날 수 있다**고 가정하고 설계를 한다. 문제에 1000 이하라고 적혀 있어도, 이 로직이 100만 개에서도 돌아갈 로직인가?를 보기 위해 N을 변수로 남겨두는 관례가 있음!

> 2️⃣ M은 N이 커져도 변하지 않는 상수이기 때문이다.
>
만약 선수가 1000명에서 1000000명으로 늘어난다고해도 팀 번호 제한은 여전히 200개이다.
이 경우 N은 실행 시간에 비례해서 영향을 주지만, 정렬해야 할 팀의 개수(200)는 더 이상 늘어나지 않고 고정되어 있져? 따라서 N에 비해서 **영향력이 고정된** M을 상수취급 하는 것이다!


이론적으로 N < 1000이라는 제약이 절대적이라면 이 문제는 어떤 경우에도 1초를 넘지 않으므로 O(1)이라고 해도 수학적으로 틀린 건 아니다.
하지만 그렇게 하면 **효율적인 로직**과 **멍청한 로직**을 구분할 수가 없게 됨.

* 로직 A (O(N)): 1,000번 연산
* 로직 B (O(N^2)): 1,000,000번 연산

둘 다 N < 1000 안에서는 O(1) 상수 시간 이지만, 로직 A가 훨씬 좋은 건 사실이잖아요? 이 차이점(효율성)을 설명하기 위해 N이라는 문자를 빌려와서 표기하는 것!

> ✚ N이 1000 미만이니 O(1)이다! 라고 한다면 **데이터 규모에 따른 확장성(Scalability)** 을 고려하지 않은 것

>💡 만약 팀 개수도 1 < M < N 이었다면 M도 변수로 취급되었을 것!!! N에 따라 변화하는 변수니까!
